# Spiral 3: Hardened Cryptography & Passive Security

**Phase:** Advanced Development Cycle
**Theme:** Military-Grade Encryption, Key Derivation, and Data Leakage Protection

---

## 1. Executive Summary & Technical Scope
Spiral 3 was the most "Security-Heavy" phase of the project. The team recognized that steganography (hiding) is not a substitute for cryptography (locking). We implemented a "Double-Layer" defense: first, the message is encrypted using AES-256-GCM; second, the resulting ciphertext is hidden using LSB scattering. Furthermore, we introduced "Passive Defense" mechanisms to protect the data even after it is revealed to the user.

### Technical Goals:
1.  **Authenticated Encryption:** Implement AES-GCM (Galois/Counter Mode) for both privacy and integrity.
2.  **Key Hardening:** Use PBKDF2 with 100,000 iterations to turn a simple passphrase into a high-entropy key.
3.  **Clipboard Guardian:** Automatically wipe the system clipboard to prevent "ghost" leaks of the secret message.

---

## 2. Detailed Technical Breakdown

### 2.1 AES-256-GCM vs. AES-CBC
The team chose **AES-GCM** because it provides "Authenticated Encryption." In standard AES-CBC, an attacker can modify the hidden bits to change the message without the user knowing. In AES-GCM, the algorithm produces a "Tag." If even one bit is changed by an attacker, the decryption fails immediately, alerting the user that the image has been tampered with.

### 2.2 PBKDF2 Key Derivation
Passphrases like "password123" are easy to crack. We implemented **PBKDF2** (Password-Based Key Derivation Function 2). 
- **Process:** We take the passphrase, add a salt (`stego_salt_fixed_16b`), and run it through 100,000 rounds of HMAC-SHA256. 
- **Result:** It takes a modern computer significantly longer to "brute force" a PBKDF2 key compared to a simple hash, providing superior protection against dictionary attacks.

### 2.3 Passive Clipboard Guardian
A major security flaw in mobile apps is the clipboard. If a user decodes a secret and copies it, it stays in the clipboard indefinitely, accessible by any other app.
- **Logic:** After a successful decode, the app starts a 60-second `Timer`. 
- **Wipe:** After 60 seconds, the app checks if the clipboard still contains the secret. If it does, it overwrites it with an empty string, effectively "shredding" the data.

## 3. Team Contributions & Work Breakdown

### Member 1: Cryptography Architect (Security Core)
**Focus:** AES Implementation & Key Logistics
- Configured the `CryptoService.dart` using the `encrypt` and `pointycastle` packages.
- Implemented the "Combined Payload" logic: [12 bytes IV] + [Ciphertext] + [16 bytes Tag]. This ensures all cryptographic components are stored together inside the image bits.
- Researched and implemented the PBKDF2 stretching algorithm to ensure even weak passphrases provided strong encryption keys.
- **Key Deliverable:** A "Zero-Knowledge" encryption engine where the app developer has no way to recover user messages.

### Member 2: Security Integration Specialist (Middleware)
**Focus:** Data Leakage & Background Services
- Developed the `ClipboardGuardian` logic. This required careful handling of the `Clipboard` API and `Timer` lifecycles to prevent memory leaks.
- Integrated the "Authentication Failure" logic: if decryption fails (wrong password), the app now reports "Authentication Failed" instead of showing garbled text.
- Added "Salt Management" to ensure consistent key derivation across different device installs.
- **Key Deliverable:** The 60-second Passive Auto-Wipe system, a standout security feature for the demo.

### Member 3: UI Designer (Hardened UX)
**Focus:** Visual Security & Anti-Selection
- Redesigned the "Decode Results" box. We moved away from standard text inputs to a "Hardened Text View" that disables all context menus.
- Removed all "Copy to Clipboard" buttons to encourage a "Read-and-Remember" security philosophy.
- Implemented "Masked Inputs" for the passphrases, including a "Visibility Toggle" that defaults to hidden for privacy during demonstrations.
- **Key Deliverable:** A "No-Leaking" UI that prevents accidental copy-pasting or screen-reading by malicious background apps.

### Member 4: Quality & Compliance Lead (Audit)
**Focus:** Entropy Analysis & Stress Testing
- Conducted "Bit-Density" audits to ensure that the added overhead of AES-GCM (IV + Tag) didn't cause "Visual Artifacts" in smaller images.
- Managed the "Library Stability" report, verifying that the `encrypt` package worked consistently on both ARM (physical) and x86 (emulator) architectures.
- Performed "Interference Testing": verified that the steganography delimiter (`##STEGO_END##`) didn't clash with the random bytes generated by the AES cipher.
- **Key Deliverable:** The Cryptographic Audit Report, proving the math behind the "Zero-Trust" model.

## 4. Challenges & Engineering Solutions

| Challenge | Engineering Solution |
| :--- | :--- |
| **Ciphertext Expansion** | Encrypted text is longer than plaintext. We updated the `CapacityCalculator` to subtract 28 bytes (IV + Tag) from the total available space. |
| **Passphrase Sensitivity** | Even a space at the end of a password changes the key. we added `.trim()` logic to the password fields to prevent user error. |
| **Clipboard Overwriting** | Some Android skins (like MIUI) block clipboard access. We implemented a "User Notification" snackbar to confirm when the wipe was successful. |

## 5. Evaluation of Spiral 3
Spiral 3 turned StegoVault into a professional-grade security tool. We moved beyond "hiding" into "protecting." The app is now resistant to both steganalysis (detecting the hidden data) and cryptanalysis (cracking the hidden data).

**Conclusion:** The final step is to lock the entire app behind hardware biometrics and a "Stealth" identity in Spiral 4.
